# CrushTask Web MVP PRD (中部重构版)

## 产品定位（不变）

**CrushTask**：以 AI 为核心、以产品功能为组织方式的任务执行系统。
- 左侧：产品功能任务树
- 中部：单个任务的全生命周期控制台（定义 → 上下文 → AI执行 → 验收）

---

## 中部区域重构需求（核心变更）

### 目标
将中部从"文档式文本展示"改为"模块化属性面板"，让任务呈现为一个可操作、可编辑、可被AI理解的对象，而不是一段说明文字。

### 结构（垂直分段、模块化、可折叠）

中部分为 4 个独立模块块，从上到下依次排列：

#### 1. 任务目标模块
- 标题：任务目标
- 内容：单行/多行可编辑文本
- 样式：独立区块、可折叠

#### 2. 输入输出模块
- 标题：输入 / 输出
- 内容：结构化展示（输入：xxx；输出：xxx）
- 可编辑
- 独立区块、可折叠

#### 3. 约束条件模块
- 标题：约束条件
- 内容：列表式（• 项1 • 项2）
- 可编辑
- 独立区块、可折叠

#### 4. 验收标准模块
- 标题：验收标准
- 内容：可勾选清单（☐ 项1 ☐ 项2）
- 可编辑
- 独立区块、可折叠

### 整体风格

- 简洁、专业、深蓝主题
- 模块之间有明显分隔
- 每个模块可独立折叠/展开
- 保持现有标签页（任务本体 / 上下文 / AI执行 / 验收）不变
- 只重构"任务本体"标签页内的内容展示方式

## 5. 开发流程

### 5.1 总体原则

- 以PRD为唯一开发基准，所有功能迭代、UI调整、代码实现均需遵循文档定义
- 以产品功能任务树为开发单元，按「项目→模块→子功能→原子任务」层级推进
- 以AI为核心执行器，构建「定义→AI执行→验收」的闭环开发模式
- 严格遵循「先更新PRD→再生成UI→再开发实现→再验收确认」的流程

### 5.2 标准开发流程（6步闭环）

1. **需求确认**：明确迭代模块与范围，更新PRD对应功能描述、UI规范、验收标准
2. **UI设计**：基于更新后的PRD生成UI结构确保左侧任务树、中部任务详情符合规范
3. **AI开发**：AI依据PRD与UI结构完成代码生成、功能实现、交互开发
4. **功能验收**：对照PRD验收标准逐项校验，通过则标记完成，不通过则反馈修改
5. **版本迭代**：单次迭代聚焦单一模块/任务，迭代后更新PRD版本号
6. **文档同步**：所有开发修改同步至PRD，保持文档与产品一致

### 5.3 角色与交付物

- **角色**：产品负责人（需求定义、PRD维护、验收）；AI（代码生成、UI实现、开发执行）
- **交付物**：更新后的PRD、UI结构图、可运行代码、验收记录

### 5.4 约束规则

- 禁止未更新PRD直接开发
- 禁止脱离任务树结构开发
- 禁止中部任务详情出现子节点/流程图
- 禁止回归文档式文本的UI形态
